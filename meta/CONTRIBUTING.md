# Contributing to the Mustadem Knowledge Base

Welcome to the Mustadem knowledge base repository! This document provides guidelines for contributing to our documentation and resources.

## How to Contribute

### Getting Started

1. **Familiarize yourself with the repository**
   - Review the [Repository Structure](./repo_structure_explained.md) to understand how content is organized
   - Read existing documents in your area of interest to understand style and depth

2. **Identify contribution opportunities**
   - Content gaps or outdated information
   - Errors or inconsistencies
   - New topics aligned with company services
   - Improvements to existing documentation

3. **Propose your contribution**
   - For minor edits: Submit directly via pull request
   - For major additions: Open an issue first to discuss scope and approach

### Contribution Types

**Content Updates**
- Correcting factual errors
- Updating statistics and market data
- Adding clarifications
- Fixing formatting issues

**New Content**
- New topic documents
- Case studies and examples
- Templates and checklists
- Reference materials

**Structural Improvements**
- Better organization
- Improved navigation
- Cross-linking between documents
- Index and glossary additions

## Documentation Standards

### Writing Style

**Tone and Voice**
- Professional yet accessible
- Technical accuracy without unnecessary jargon
- Practical and actionable guidance
- Consistent with Saudi Arabian business context

**Language**
- Primary language: English
- Use clear, concise sentences
- Define technical terms on first use
- Avoid colloquialisms and regional expressions

**Formatting Guidelines**
- Use Markdown formatting consistently
- Maximum 100 characters per line for readability
- Use headings hierarchically (H1 → H2 → H3)
- Include table of contents for documents over 500 lines

### Content Requirements

**Technical Accuracy**
- All technical content must be verifiable against industry standards
- Include sources for statistics and market data
- Use current data (2024-2025 preferred)
- Validate calculations and formulas

**Regional Relevance**
- Prioritize Saudi Arabia and GCC context
- Use SAR (Saudi Riyal) for currency with USD conversion at 3.75 rate
- Reference local regulations and government programs
- Consider desert climate and regional conditions

**Practical Application**
- Include specific numbers and ranges
- Provide actionable checklists and templates
- Reference real-world scenarios
- Avoid placeholder or TBD content

### Document Structure

**Standard Document Template**
```markdown
# Document Title

## Overview
Brief introduction and purpose

## Main Content Sections
Organized with clear headings

## Practical Application
Actionable guidance, templates, examples

## Related Documents
Cross-references to related content

---

**Document Information:**
- Version: X.X
- Last Updated: Month Year
- Review Schedule: Frequency
```

**Required Elements**
- Clear, descriptive title
- Purpose statement or overview
- Logical section organization
- Cross-references where relevant
- Version and date information

### Formatting Standards

**Headings**
```markdown
# H1 - Document Title (one per document)
## H2 - Major Sections
### H3 - Subsections
#### H4 - Detailed Topics
```

**Lists**
- Use bullet points for unordered lists
- Use numbered lists for sequences or rankings
- Maintain consistent list formatting throughout

**Tables**
```markdown
| Column 1 | Column 2 | Column 3 |
|----------|----------|----------|
| Data     | Data     | Data     |
```

**Code/Technical Content**
- Use code blocks for formulas, commands, or technical specifications
- Include explanatory comments where helpful

**Links**
- Use relative paths for internal links: `[Link Text](../folder/file.md)`
- Clearly label external links
- Verify all links work before submission

## Review Process

### Submission Process

1. **Create a branch** from main for your changes
2. **Make your edits** following documentation standards
3. **Self-review** using the checklist below
4. **Submit pull request** with clear description of changes
5. **Address feedback** from reviewers

### Self-Review Checklist

Before submitting, verify:

- [ ] Content is technically accurate
- [ ] Writing follows style guidelines
- [ ] Formatting is consistent with existing documents
- [ ] All links work correctly
- [ ] No placeholder or TBD content remains
- [ ] Cross-references are appropriate
- [ ] Version information is updated

### Review Criteria

Reviewers will evaluate:

**Accuracy**
- Technical correctness
- Current and valid data sources
- Proper calculations

**Quality**
- Clear and professional writing
- Appropriate depth and detail
- Practical applicability

**Consistency**
- Alignment with existing content
- Proper formatting
- Correct cross-referencing

**Relevance**
- Alignment with company focus areas
- Value to intended audience
- Appropriate scope

### Review Timeline

- **Minor edits:** 1-2 business days
- **New content:** 3-5 business days
- **Major restructuring:** 5-10 business days

## Code of Conduct

### Professional Standards

All contributors agree to:

1. **Maintain professionalism** in all communications
2. **Respect intellectual property** and cite sources appropriately
3. **Protect confidentiality** of client and company information
4. **Accept feedback constructively** and engage respectfully
5. **Support fellow contributors** in improving the knowledge base

### Content Guidelines

**Do:**
- Provide accurate, verified information
- Use inclusive, professional language
- Respect copyright and attribution requirements
- Focus on practical, actionable content

**Don't:**
- Include confidential client information
- Copy content without proper attribution
- Use discriminatory or unprofessional language
- Promote personal or commercial interests

### Conflict Resolution

If disagreements arise:

1. Discuss concerns respectfully in comments
2. Seek input from additional reviewers
3. Escalate to repository maintainers if needed
4. Accept final decisions gracefully

## Version Control Guidelines

### Commit Messages

Use clear, descriptive commit messages:
- `Add new SOP for water quality testing`
- `Update Saudi market statistics to 2025 data`
- `Fix formatting in financial model template`

### Branch Naming

Use descriptive branch names:
- `feature/new-sop-template`
- `update/market-data-2025`
- `fix/broken-links-project-delivery`

### Pull Request Descriptions

Include:
- Summary of changes
- Reason for changes
- Any related issues or dependencies
- Testing or verification performed

## Getting Help

### Questions and Support

- **Documentation questions:** Open an issue with "Question:" prefix
- **Technical assistance:** Contact repository maintainers
- **Process guidance:** Review this document and related resources

# Contributing to Mustadem Resources

Thank you for your interest in contributing! This guide will help you contribute effectively.

## How to Contribute

### Types of Contributions
1. **Content Updates** - Corrections and improvements
2. **New Content** - Templates, guides, documentation
3. **Translations** - Arabic translations and reviews
4. **Structure Improvements** - Better organization

## Guidelines

### Content Standards
- **Clear and Concise**: Plain language
- **Actionable**: Specific steps and examples
- **Accurate**: Verified technical information
- **Current**: Up-to-date data

### Bilingual Requirements
- Both versions must match in structure
- Technical accuracy in both languages
- Culturally appropriate examples

### File Organization
```
category/
  ├── en/document.md
  └── ar/document_ar.md
```

## Contribution Process

1. Plan your content
2. Write English version
3. Create Arabic version
4. Review both versions
5. Submit contribution

## Quality Checklist
- [ ] Content is accurate
- [ ] Both language versions exist
- [ ] Formatting is consistent
- [ ] Links work
- [ ] Examples included
- [ ] Sources cited
